## 11.1 Defaults (Zero Config)

A Kanon project requires no configuration by default. The toolchain infers everything from conventions:

- **Entry point:** `main.kan` in the current directory
- **Local modules:** Resolved by filesystem path relative to the project root (see §9)
- **External dependencies:** Fetched from git URLs on first import, pinned in `kanon.lock`

```bash
# single-file script — just run it
kanon run sketch.kan

# multi-file project — runs main.kan
kanon run
```

A project can be as simple as one file with no config, no manifest, no lockfile. Complexity is added only when needed.

## 11.2 Lockfile (`kanon.lock`)

**Auto-generated. Never hand-edited. Commit to version control.**

Created automatically the first time an external dependency is fetched. Records the exact resolved commit for each external import.

```toml
# kanon.lock — auto-generated by kanon toolchain

[deps."github.com/someone/xenharmonic-scales"]
url = "https://github.com/someone/xenharmonic-scales.git"
commit = "a1b2c3d4e5f6"
fetched = "2026-03-15T10:30:00Z"

[deps."github.com/other/cool-patterns"]
url = "https://github.com/other/cool-patterns.git"
commit = "f6e5d4c3b2a1"
fetched = "2026-03-20T14:00:00Z"
```

**Purpose:** Reproducible builds. Your collaborators and CI get the same dependency versions you do.

```bash
kanon update                              # re-fetch all deps, update lockfile
kanon update "github.com/someone/repo"    # update one dep
```

The lockfile works with or without a config file. Without a config, it pins whatever HEAD was at first fetch.

## 11.3 Config (`kanon.toml`) — Optional

Human-authored. Only needed when the toolchain's defaults aren't sufficient.

```toml
[project]
name = "my-composition"
version = "0.1.0"
entry = "main.kan"        # only needed if entry point isn't main.kan

[deps."github.com/someone/xenharmonic-scales"]
branch = "main"

[deps."github.com/other/cool-patterns"]
tag = "v1.2.0"
```

**When you need it:**
- Entry point isn't `main.kan`
- You want to pin a dependency to a specific branch or tag (rather than just HEAD)
- Project metadata for publishing (name, version, author)
- Future: host/platform-specific configuration

**When you don't need it:** Single-file scripts, multi-file projects that follow conventions, projects with no external dependencies or no version constraints.

## 11.4 Relationship Between Config and Lockfile

| | Config (`kanon.toml`) | Lockfile (`kanon.lock`) |
|---|---|---|
| **Created by** | Human | Toolchain |
| **Contains** | Intent (constraints, metadata) | Facts (resolved commits) |
| **Edited by** | Human | `kanon update` command |
| **Required** | No | Auto-created when needed |
| **Commit to VCS** | Yes | Yes |

The config says *what you want*. The lockfile says *what you got*. Without a config, the lockfile still works — it just pins whatever was fetched.

## 11.5 Project Layout Examples

**Minimal (single file):**
```
sketch.kan
```

**Simple (multi-file, no config):**
```
main.kan
scales.kan
patterns.kan
```

**With external deps (auto-generated lockfile):**
```
main.kan
scales.kan
kanon.lock           ← auto-generated
```

**Full project (config + lockfile):**
```
kanon.toml           ← optional, human-authored
kanon.lock           ← auto-generated
main.kan
scales.kan
tuning.kan
patterns/
    arpeggios.kan
    rhythms.kan
```

## 11.6 Further Reading

This design follows established patterns. For deeper background:

- **Go modules:** [go.dev/blog/using-go-modules](https://go.dev/blog/using-go-modules) — the lockfile (`go.sum`) and manifest (`go.mod`) split
- **Cargo (Rust):** [doc.rust-lang.org/cargo](https://doc.rust-lang.org/cargo/) — `Cargo.toml` (manifest) vs `Cargo.lock` (lockfile)
- **Deno:** [deno.land](https://deno.land) — URL-based imports with a lockfile, no package.json by default. Closest to Kanon's "zero config + git URLs" approach.
